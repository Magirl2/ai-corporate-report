# 데이터 캐싱 및 Redis 관리 (Redis & Cache)

## 목적
데이터 조회 속도를 높이고 외부 API 호출 비용을 절감하기 위해 Redis 기반의 캐싱 시스템을 효율적으로 관리하기 위함입니다.

## 언제 사용하나
- 대규모 데이터(DART, 보고서 등)를 캐싱하거나 조회할 때
- 사용자 사용량 제한(Rate Limit) 정보를 관리할 때
- 세션이나 중간 연산 결과를 임시 저장할 때

## 관련 파일
- `api/_lib/db.js`
- `api/_lib/handlers/report/cache.js`
- `api/_lib/handlers/user/usage.js`
- `.gitignore`

## 핵심 규칙
- 운영 환경에서는 로컬 파일 시스템 대신 Upstash Redis 등 클라우드 KV 저장소를 사용한다.
- 캐시 데이터의 특성에 맞춰 적절한 TTL(Time To Live)을 설정한다.
- 민감한 데이터는 캐싱 전후로 보안 처리를 고려한다.
- `.users.json`, `.reports_cache.json` 등 로컬 테스트용 캐시 파일이 Git에 추적되지 않도록 관리한다.
- 캐시 Hit/Miss 상황을 로그로 남겨 효율성을 모니터링한다.

## 검증 방법
- `git status`를 실행하여 로컬 캐시 파일이 포함되지 않았는지 확인
- 회원가입 후 사용자 정보가 DB(Redis)에 정상적으로 저장/조회되는지 확인
- 보고서 생성 시 캐시 데이터가 정상적으로 생성되고 활용되는지 확인

## 흔한 실수
- 캐시 TTL을 너무 길게 설정하여 업데이트된 정보가 반영되지 않는 경우
- Redis 연결 실패 시에 대한 예외 처리가 없어 전체 API가 중단되는 경우
- 로컬 개발 환경의 캐시 데이터를 그대로 프로덕션으로 옮기려 하는 경우
