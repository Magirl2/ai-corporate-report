# 백엔드 API 설계 및 에러 처리 (Backend API)

## 목적
표준화된 API 구조와 일관된 에러 응답을 통해 프론트엔드와의 협업 효율성을 높이고 유지보수성을 확보하기 위함입니다.

## 언제 사용하나
- 새로운 API 엔드포인트를 추가할 때
- 기존 API의 응답 형식을 변경하거나 기능을 확장할 때
- 전역적인 에러 핸들링 로직을 개선할 때

## 관련 파일
- `api/_lib/handlers/**`
- `api/_lib/errors.js`
- `api/_lib/logger.js`
- `api/_lib/orchestrator.js`
- `api/_lib/env.js`

## 핵심 규칙
- Vercel API Route 구조에 맞춰 파일 및 폴더를 배치한다.
- 요청 메서드(GET, POST 등)를 엄격히 검증한다.
- 모든 응답은 일관된 JSON 형식을 유지한다.
- HTTP 상태 코드를 표준에 맞춰 사용한다 (200, 201, 400, 401, 404, 500 등).
- 에러 응답 시 `code`, `message`, `details`를 분리하여 제공한다.
- 서버 내부의 기술적인 오류와 사용자에게 보여줄 메시지를 분리한다.
- 서버리스 타임아웃을 고려하여 응답 속도를 최적화한다.

## 검증 방법
- API 직접 호출을 통한 상태 코드 및 응답 스키마 확인
- 민감 정보가 응답 데이터에 포함되었는지 확인
- `npm run build`를 통한 문법 및 의존성 검사

## 흔한 실수
- 모든 에러를 200 OK로 반환하고 JSON 내부 상태값으로만 구분하는 경우
- 요청 본문(Body) 데이터의 유효성 검사를 누락하는 경우
- 비동기 작업(`await`) 처리를 누락하여 요청이 완료되기 전 응답하는 경우
