# LLM 프롬프트 및 지능형 분석 (Prompt & LLM)

## 목적
LLM의 성능을 극대화하고 할루시네이션을 방지하며, 정교한 분석 리포트를 생성하기 위한 프롬프트 전략을 수립하기 위함입니다.

## 언제 사용하나
- 새로운 분석 에이전트의 프롬프트를 작성하거나 수정할 때
- 보고서의 논리적 구조나 문체, 근거 제시 방식을 개선할 때
- 모델 응답의 일관성과 형식을 강제해야 할 때

## 관련 파일
- `api/_lib/prompts.js`
- `api/_lib/orchestrator.js`
- `api/_prompts/report-agents/` (모든 런타임 프롬프트 파일)
- `resolver.md`, `analyst-*.md`, `critic.md`, `composer.md`

## 핵심 규칙
- `orchestrator.js`에서 호출하는 `agentName`과 프롬프트 파일명을 반드시 일치시킨다.
- 모든 프롬프트는 JSON 출력을 기본으로 하며, 필요한 경우 엄격한 Schema를 제공한다.
- Gemini 모델 등 사용 가능한 모델의 특성을 고려한 프롬프트를 작성한다.
- 할루시네이션 방지를 위해 제공된 데이터(`context`) 내에서만 답변하도록 강제한다.
- 모든 주장에는 `sourceId`를 병기하여 근거를 명확히 한다.
- 신뢰도가 낮은 소스는 단정적인 결론의 근거로 사용하지 않도록 지시한다.
- 투자 권유나 직접적인 매수/매도 추천 등 법적 문제가 될 수 있는 표현을 금지한다.
- 토큰 예산과 타임아웃을 고려하여 응답의 길이를 조절한다.

## 검증 방법
- 생성된 Markdown 리포트의 논리 구조 및 근거 제시 적절성 확인
- `sourceQualitySummary` 내용이 `composer` 결과물에 반영되었는지 확인
- `npm run build` 성공 확인

## 흔한 실수
- 프롬프트에서 데이터 형식을 명확히 지정하지 않아 파싱 에러가 발생하는 경우
- "인터넷 정보를 찾아봐라"와 같이 제어 불가능한 외부 검색을 지시하는 경우 (수집된 컨텍스트만 활용해야 함)
- 특정 종목에 대해 과도하게 긍정적이거나 부정적인 편향을 보이는 경우
