# 공통 개발 원칙

- 먼저 관련 파일을 읽고 현재 코드 기준으로 판단한다.
- 추측하지 말고 코드와 실행 결과를 근거로 보고한다.
- 기존 기능을 깨지 않는 최소 변경을 우선한다.
- 변경 전후 파일 목록을 명확히 보고한다.
- npm run build를 실행하고 결과를 보고한다.
- 필요한 경우 git diff, git grep, Select-String, findstr 등으로 검증한다.
- API key, JWT secret, Redis token 등 민감값은 코드/로그/응답에 절대 노출하지 않는다.
- 환경변수 누락 시 안전한 오류 메시지를 반환한다.
- 사용자에게 보이는 에러 메시지와 서버 내부 로그를 분리한다.
- UI 변경은 모바일 반응형과 터치 입력을 고려한다.
- 빈 데이터, 오류, 로딩, 부분 성공 상태를 반드시 처리한다.
- TODO, placeholder, “나중에 추가” 같은 미완성 문구를 남기지 않는다.
- Vercel 배포 환경과 serverless timeout을 고려한다.
- 보고서 생성 파이프라인은 Stage 1/2/3 실패를 구분한다.
- 종합 보고서 실패가 개별 섹션 표시를 막지 않도록 partial success를 유지한다.
- 출처와 데이터 신뢰도를 사용자가 확인할 수 있게 표시한다.

# 검증 체크리스트

모든 작업 후 가능한 범위에서 아래를 확인한다.

- npm run build 성공
- 관련 API 직접 호출 확인
- 민감값 grep 검색
- 주요 문자열 grep 검색
- UI 변경 시 모바일 레이아웃 확인
- 기존 로그인/회원가입/검색/보고서 생성 흐름 유지
- 변경 파일 목록과 핵심 변경 요약 보고
